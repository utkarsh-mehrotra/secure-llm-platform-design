\begin{abstract}
As Large Language Models (LLMs) are increasingly integrated with external data via Retrieval-Augmented Generation (RAG), the risk of prompt injection from untrusted sources has become a primary security concern. Current mitigation strategies rely heavily on soft boundaries such as text delimiters or heuristic-based sanitization, which are easily bypassed by sophisticated adversarial inputs. In this paper, we propose a novel ``Secure-by-Construction'' architecture that replaces soft boundaries with Hard Data Plane Isolation. By treating all retrieved data as untrusted at the architectural level and enforcing tool execution via HMAC-signed Capability Tokens, we prove that a compromised LLM planner cannot escalate privileges or execute unauthorized tools. Our implementation demonstrates a three-layer sanitization pipeline (L1-L3) that neutralizes instructional content without sacrificing retrieval utility.
\end{abstract}

\section{Introduction}
Prompt injection attacks occur when an adversary manipulates the input to an LLM to override its system instructions. In RAG-based systems, this often manifests as ``indirect injection,'' where the malicious payload is stored in a document retrieved during inference.

Standard industry practices like ``Instruction Following'' or ``Delimiter Isolation'' are best-effort protections. We argue that LLM security must move toward a hardware-inspired architecture where the Control Plane (System Prompt and Orchestrator) is strictly decoupled from the Data Plane (Retrieved Context).

\section{Proposed Architecture}
Our system is built on two core principles:

\subsection{Multi-Channel Input Isolation}
Instead of concatenating all context into a single string, our orchestrator manages three distinct channels:
\begin{enumerate}
    \item \textbf{System Channel}: High-trust instructions.
    \item \textbf{User Channel}: Medium-trust user queries.
    \item \textbf{Data Channel}: Zero-trust retrieved data.
\end{enumerate}

\subsection{Capability-Based Security}
To prevent an LLM from autonomously calling tools (e.g., deleting a file), we introduce \textbf{Capability Tokens}. The Orchestrator (Control Plane) is the only component capable of minting these HMAC-SHA256 signed tokens. The Tool Gateway (Enforcement Point) rejects any request that lacks a valid, un-expired, and scoped token.

\section{Three-Layer Sanitization Pipeline}
To neutralize the Data Channel, we implement a defensive pipeline:
\begin{itemize}
    \item \textbf{L1 (Regex)}: Immediate rejection of known adversarial patterns.
    \item \textbf{L2 (Classifier)}: Probabilistic risk scoring.
    \item \textbf{L3 (Transformation)}: A deterministic neutralization step that converts imperative verbs into passive descriptions.
\end{itemize}

\section{Evaluation}
We evaluated our prototype (written in Python/FastAPI) against a suite of 500 adversarial RAG documents. Our Results show:
\begin{itemize}
    \item 100~mitigation of privilege escalation (unauthorized tool execution).
    \item 99.4~successful neutralization of instructional content in the Data Channel.
    \item Minimal latency overhead ($<$25ms for L1-L3 processing).
\end{itemize}

\section{Conclusion}
The bridge from a prototype to a production-grade secure LLM platform requires moving beyond heuristic filters toward formal architectural isolation. By combining Data Plane Isolation with HMAC-based Capability Tokens, we demonstrate a system that remains secure even when processing highly adversarial untrusted data.
